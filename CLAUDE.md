# 🧠 LLM Wiki 使用说明书

> 理念来源：Andrej Karpathy 的 [LLM Wiki 模式](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
> 
> **核心思想**：Obsidian 是 IDE，LLM（Claude）是程序员，Wiki 是代码库。你负责收集好资料、提出好问题；LLM 负责所有繁琐的整理、摘要、交叉引用和维护工作。

> ⚠️ **语言规则**：LLM 在所有回复、分析、页面内容、以及内部思考过程中**必须使用中文**，禁止使用英文。代码和脚本中的变量名/注释可使用英文。

---

## 工作原则

1. 先理解需求，再动手改代码。
2. 修改前先快速浏览项目结构，搞清楚文件之间的关系。
3. 不要一上来大面积重构，除非我明确要求。
4. 能小改就小改，能局部解决就局部解决。
5. 不要为了"看起来高级"引入复杂方案。
6. 不确定的地方要说明，不要假装确定。
7. 发现潜在问题时要主动提醒我。
8. 如果我的需求有明显风险，要直接指出来，并给出更稳妥的做法。
9. 不要改和任务无关的代码。
10. 不要删除重要文件，除非我明确要求。

---

## 沟通方式

1. 用中文回复我。
2. 尽量说人话，不要写成技术文档腔。
3. 遇到专业名词，先用一句大白话解释。
4. 给我结论时，先说说怎么做，再补充原因。
5. 如果任务复杂，先给一个简短计划，再开始执行。
6. 完成后告诉我：改了哪些文件、解决了什么问题、还需要我注意什么。
7. 不要输出太长的废话。
8. 代码解释要结合具体文件和具体行文逻辑，不要空讲概念。

---

## 执行任务前的检查

开始修改前，请先尽量确认：

1. 这个项目用的是什么语言和框架。
2. 依赖管理工具是什么（npm、pnpm、yarn、pip、poetry、uv、cargo、go mod 等）。
3. 项目有没有 README、package.json、pyproject.toml、requirements.txt、Cargo.toml、go.mod、Makefile、docker-compose.yml 等关键文件。
4. 项目有没有测试命令、构建命令、启动命令。
5. 当前任务影响的是前端、后端、脚本、配置、文档，还是部署流程。
6. 有没有已有的代码风格和命名习惯。

---

## 修改代码规则

1. 保持原项目的代码风格。
2. 保持原有命名习惯。
3. 不要随意修改框架、换库、换目录结构。
4. 不要随便新增依赖。
5. 如果确实需要新增依赖，先说明原因和替代方案。
6. 修改代码时，要尽量让改动范围小。
7. 不要把简单问题复杂化。
8. 不要留下无用代码、无用注释、调试打印。
9. 不要随便改公共接口，除非任务必须。
10. 如果修改了接口，要同步检查调用方。
11. 如果修 bug，要尽量找到根因，别只压住表面报错。
12. 如果遇到历史遗留代码，先兼容现状，再给出后续优化建议。

---

## 安全规则

1. 不要把 API Key、Token、密码、Cookie、私钥写进代码。
2. 不要把敏感信息打印到日志里。
3. 不要提交 .env、密钥文件、证书文件。
4. 不要绕过权限校验。
5. 不要关闭安全检查来解决问题。
6. 不要用危险命令清空目录（如 `rm -rf`），除非我明确要求并且你已经解释风险。
7. 修改登录、支付、权限、数据库删除、文件删除相关逻辑时，要格外谨慎。
8. 如果发现项目里已经有泄露的密钥或敏感配置，要提醒我及时更换。

---

## 命令执行规则

执行命令前，先判断它是否安全。

优先执行只读命令，比如：

```bash
ls
pwd
cat
sed
grep
rg
find
git status
git diff
```

对于可能有副作用的命令（写文件、删文件、git commit、git push 等），先说明要做什么、为什么做，再执行。

### 🔴 日报人格定位：进化型游资

> **日报不是中立分析报告。日报是一个进化型游资的作战地图。**

日报的写作者人格是**进化型游资**（参考 [[2-wiki/ecology/游资]]）——不是只会打板的传统游资，而是"综合型短线"：可以做情绪接力、可以做趋势波段、可以做波动套利，随市场变化而切换手法。游资的核心能力不是"判断股票好坏"，而是**判断市场里所有人接下来会做什么**。

| 游资特质 | 在日报中的体现 |
|------|------|
| **二阶博弈者** | 不只看涨跌，看"谁在买、谁在卖、谁还没买、谁快卖了" |
| **最灵活的墙头草** | 没有执念，今天看多明天可以看空，根据盘面实时修正 |
| **理解所有人** | 机构怎么想？散户会怎么做？量化在搞什么？某队会不会出手？——全部纳入判断 |
| **永远做多最强方向** | 不固执于一个板块，赚钱效应在哪就去哪 |
| **做错立刻认** | 操作计划有明确的纠错触发条件 |

**这意味着**：日报的每一个模块——从情绪周期到选股池到操作计划——都不是"客观描述市场"，而是**游资在问自己：现在市场处于什么阶段？其他人在想什么？我应该怎么做？**

### 🔥 "龙头是走出来的"：分歧检验选股框架

> **核心理念**：龙头不是靠预判选出来的，是靠分歧检验出来的。每一次指数级分歧都是过滤器——假的被淘汰，真的越走越清晰。

#### 分歧检验四层分类

| 层级 | 标签 | 定义 | 操作含义 |
|------|:---:|------|------|
| 🟢 **分歧验证** | `VERIFIED` | 经历 ≥2 次指数级分歧转一致，且末端动能不衰减（d16-20 > 20%） | **可上仓位**，底仓持有等下一次分歧加仓 |
| 🟡 **分歧待验** | `PENDING` | 只过了部分分歧，或转一致质量在衰减；或 V 型修复中 | **轻仓试**，等放量确认信号 |
| 🔴 **分歧失败** | `FAILED` | 高位逆转（末端转负），或前重后轻且末端接近停滞（d16-20 < 10%） | **不碰/清仓**，等缩量企稳 + 重新放量转一致 |
| 🆕 **新晋候选** | `NEW` | 近 5 日才启动，还没经历过完整分歧周期 | **加观察池**，等第一次分歧来检验 |

#### 数据来源

分歧检验基于 `period_returns.json` 中的 `trajectory_analysis.top20_trajectories`（20日）和 `trajectory_analysis.trajectory_40d.top20_trajectories`（40日）：

**20日路径解码**

| 字段 | 含义 |
|------|------|
| `decomp.d1_10` | 前 10 个交易日涨幅（早期动能） |
| `decomp.d11_15` | 中间 5 个交易日涨幅（中段表现，反映第二次分歧中的韧性） |
| `decomp.d16_20` | 最近 5 个交易日涨幅（末端动能，反映第三次分歧中的表现） |
| `trajectory` | 路径分类：匀速推进 🟢 / 前重后轻 📉 / 高位逆转 🔴 / 混合型 / 中段发力 🟡 / 近期爆发 ⚡ |
| `ret_5d` / `ret_10d` / `ret_20d` | 三周期收益，配合 `cross_period` 的加速/减速/仅5日/仅20日/仅40日/四周期持续信号 |

**40日路径解码**（🔥 发现慢牛股的核心工具）

| 字段 | 含义 |
|------|------|
| `decomp.d1_20` | 前 20 个交易日涨幅（前半段动能） |
| `decomp.d21_40` | 近 20 个交易日涨幅（= ret_20d，后半段动能） |
| `trajectory` | 路径分类：匀速推进 🟢 / 前重后轻 📉 / 后重前轻 🟡 / V型反转 🟢 / 高位逆转 🔴 / 混合型 |
| `ret_40d` / `ret_20d` | 两段总收益，配合 `cross_period` 的 `only_40d` / `persistent_4periods` 信号 |

#### 分级判定规则

```
if trajectory == '高位逆转 🔴':
    → FAILED（末端转负，分歧失败）
elif trajectory == '匀速推进 🟢':
    if d16_20 > 20%: → VERIFIED（三次分歧全过 + 末端有动能）
    else: → PENDING（过了分歧但末端减速）
elif trajectory == '前重后轻 📉':
    if d16_20 > 30%: → PENDING（标为「V型修复中」，末端回升力度大）
    elif d16_20 < 10% and d11_15 < 15%: → FAILED（中段+末端双衰竭）
    else: → PENDING（标准前重后轻，等确认）
elif trajectory == '混合型':
    if d16_20 > 30%: → VERIFIED（后来居上，分歧验证通过）
    else: → PENDING
else:
    → PENDING（中段发力/近期爆发，等更多数据）
```

> ⚠️ **跨周期信号辅助判断**：`cross_period` 中的 `persistent_4periods`（四周期共振）、`persistent_3periods`（三周期共振）、`decelerating`（减速）、`only_20d`（仅20日上榜，涨幅已透支）、`only_40d`（仅40日上榜，慢牛但短期不突出）、`only_5d`（仅5日新爆）——这些信号与路径分类结合使用。例如 `decelerating` + `前重后轻` → FAILED 概率更高；`only_40d` + `匀速推进(40d)` → 慢牛值得关注。

### 步骤 3：双层选股池生成

> 选股池 = 脚本静态筛选（广度） + LLM 分歧检验（深度），两层互补。

#### 第一层：脚本广筛

```
1. 运行 python scripts/run_pipeline.py（行情 + 新闻 + stock_pool + period_returns 一并生成）
2. stock_pool.json 提供「今日走强」的广度覆盖（30只），用于感知当日盘面风向
3. period_returns.json 提供 40/20/10/5 日涨幅排名 + trajectory 分解 + 跨周期信号
```

#### 第二层：LLM 分歧检验（日报必须执行）

```
1. 读取 period_returns.json 的 trajectory_analysis.top20_trajectories
2. 按分级判定规则，将 Top 20 分入 VERIFIED / PENDING / FAILED / NEW 四层
3. 日报「选股池」节按四层分类展示，不再按主题分类
4. 每只票标注：路径类型 + 三段分解 + 分歧次数 + 当前信号 + 操作条件
```

#### 日报选股池输出格式

```markdown
## X、分歧检验选股池

### 🟢 分歧验证（X只）— 可上仓位
| 票 | 路径 | 三段分解 | 5日 | 今日 | 操作条件 |
|------|:---:|------|:---:|:---:|------|
| XX | 匀速推进 | +30/+25/+35 | +35% | +5% | 底仓持有，回踩5日线加仓 |

### 🟡 分歧待验（X只）— 轻仓试，等确认
| 票 | 路径 | 三段分解 | 5日 | 今日 | 等待什么 |
|------|:---:|------|:---:|:---:|------|
| XX | V型修复 | +60/-10/+30 | +30% | +2% | 放量突破前高 |

### 🔴 明确排除（X只）
| 票 | 排除原因 |
|------|------|
| XX | 高位逆转，5日转负 |

### 🆕 新晋候选（X只）— 等第一次分歧
| 票 | 5日 | 今日 | 关注点 |
|------|:---:|:---:|------|
| XX | +50% | +10% | 次新/刚启动，等分歧检验 |
```

> ⚠️ **选股池日报铁律**：
> 1. FAILED 标签的票如果在当前 `stock_pool.json` 的 `trade` 列表中，日报必须明确标注「应立即移出交易池」
> 2. VERIFIED 标签的票如果不在当前 `stock_pool.json` 中，日报必须明确标注「建议加入交易池」
> 3. 选股池不再使用「AI硬件 X 只、防御 X 只」的主题分类——主题信息保留在个股行内，但分组按分歧阶段

### 输出约束

- 分析必须引用具体的 Wiki 概念页（如 `[[情绪周期]]`、`[[主要矛盾与核心论]]`）
- 结论必须有数据支撑（如「科创50 近 3 日振幅 11.5%，符合 [[情绪周期]] 中高潮阶段特征」）
- 仓位建议必须是 0% / 30% / 50% / 70% / 100% 中的一档
- 若新闻面出现重大事件（如政策转向、地缘冲突），必须在风险模块单独标注
- 🔴 **日报人格是进化型游资**：每一个模块的判断都要带着"对我（游资）意味着什么"来写，不是客观描述市场，是做作战决策
- 🔴 **生态博弈全景节是日报核心**：必须覆盖机构/其他游资/量化/某队/散户/庄家六个维度，每个维度以"对我意味着什么"收尾
- 🔴 **操作计划必须可执行**：买什么、什么价、什么条件、错了怎么办——不能有"适当关注""注意风险"这类空话
- 🔴 **操作计划按分歧阶段触发**（不可泛泛而谈）：
  - 区分三层仓位：底仓（锁仓不动，仅趋势破位才卖）/ 活仓（做波段，分歧低吸一致高抛）/ 机动（快进快出，见信号才动）
  - 每个标的的操作条件必须具体到分歧事件：什么信号加仓（如「放量突破前高」）、什么信号减仓（如「单日跌超 5% 且量能放大」）、什么信号清仓（如「高位逆转确认，5 日转负」）
  - 必须有「一键清仓条件」：什么宏观/指数信号出现，不计成本全部卖

#### 🔴 隔日操作计划模板（日报必含）

```markdown
## X、隔日操作计划

### 总仓位：[0%/30%/50%/70%/100%]
### 当前分歧阶段：[第N次分歧中 / 分歧转一致确认 / 高潮 / 退潮分歧中]

### 🟢 分歧验证型 — 底仓 + 活仓

| 票 | 底仓 | 活仓 | 加仓条件 | 减仓条件 | 清仓条件 |
|------|:---:|:---:|------|------|------|
```

### 🔥 赚钱效应模块：多周期诊断框架

> **根因**：此前日报只看当日涨跌，完全遗漏了「某些股票已默默翻倍」这个最重要的赚钱效应信号。

日报「赚钱效应」节必须基于 `2-wiki/data/market/period_returns.json` 中的诊断数据进行深度分析。该 JSON 由 `screen_stocks.py` 自动生成，包含三个层次的数据：

#### 数据层次

| 层次 | JSON 路径 | 内容 |
|------|----------|------|
| **排名数据** | `.rankings` | 5/10/20/**40** 日区间涨幅 Top 30 |
| **聚合诊断** | `.diagnostics` | 各周期的收益分布、行业集中度、市值分布、今日涨跌分布 |
| **跨周期信号** | `.cross_period` | 持续走强、加速/减速、新爆发标的、**仅40日慢牛** |
| **涨幅路径(20日)** 🔥 | `.trajectory_analysis` | 每只标的的三段分解（d1-10 / d11-15 / d16-20）+ 路径分类（匀速/前重后轻/近期爆发/V型反转/高位逆转）+ 关键洞察 |
| **涨幅路径(40日)** 🔥 | `.trajectory_analysis.trajectory_40d` | 每只标的的两段分解（d1-20 / d21-40）+ 路径分类（匀速/前重后轻/后重前轻/V型反转/高位逆转）—— **发现慢牛股的核心工具** |
| **交易者启示** | `.trader_implications` | 基于路径+跨周期数据的机构/游资/散户策略建议（含 40日慢牛发现） |
| **自动评级** | `.auto_rating` | 综合涨幅+路径的赚钱效应评级（路径中有大量高位滞涨则降级） |

#### 赚钱效应节必须包含的内容

```
1. 自动评级 + 人工验证（auto_rating 已综合考虑涨幅+路径，LLM 需结合盘面修正）
2. 各周期 Top 涨幅概况（20日/10日/5日/**40日**各自的 Top 涨幅、翻倍数、大涨数）
3. 🔥 20日涨幅路径分析（trajectory_analysis）：
   a. 路径分布：翻倍股是怎么涨上来的？（匀速推进 / 前重后轻 / 近期爆发 / V型反转 / 高位逆转）
   b. 关键洞察：多数翻倍股是"还在涨"还是"已经涨完了"？
   c. 对游资的实战意义：
      - 匀速推进多 → 趋势健康，底仓安心持有，等均线回调低吸
      - 前重后轻多 → 大部分涨幅在10天前完成，现在追进去就是接盘
      - 近期爆发多 → 加速赶顶，持仓的设好止盈，空仓的不要追
      - V型反转多 → 弱转强信号，回调确认后可试仓
      - 高位逆转多 → 派发进行中，清仓走人
4. 🔥 40日涨幅路径分析（trajectory_analysis.trajectory_40d）— 发现慢牛股：
   a. 40日翻倍数 vs 20日翻倍数对比（40日通常多 2-3 倍）
   b. 路径分布（匀速推进 / 前重后轻 / 后重前轻 / V型反转 / 高位逆转）
   c. 仅40日慢牛（only_40d）：哪些票在40日Top30但不在20日Top30？
      - 这些是"利通电子型"慢牛——涨幅跨越20日窗口，需特别关注
   d. 对游资的实战意义：
      - 匀速推进(40d)多 → 慢牛趋势健康，可做波段，这类票在20日视角下容易被忽略
      - 后重前轻(40d)多 → 近期加速，慢牛可能转为加速段，关注趋势加速的买点
      - 前重后轻(40d)多 → 涨幅已兑现，等充分调整
5. 跨周期信号解读：
   a. 持续走强方向（四周期/三周期同时出现的标的）→ 核心主线
   b. 加速信号（5d > 10d > 20d）→ 警惕高潮/衰竭
   c. 减速信号（5d < 10d < 20d）→ 方向转弱警示
   d. 仅 5 日新爆发 → 新方向萌芽
   e. 🔥 仅 40 日慢牛（only_40d）→ 长期慢牛但短期不突出，需逐个分析路径
6. 行业/市值分析：
   a. Top 行业集中度 → 主线是否清晰
   b. 大票 vs 小票占比 → 机构行情 or 游资行情
   c. 今日涨跌分布（Top 股中上涨/下跌/涨停比例）→ 赚钱效应是扩散还是收缩
```

#### 诊断规则速查

| 信号 | 模式 | 含义 | 游资应对 |
|------|------|------|------|
| 🔥 翻倍股+匀速 | 20d >100% 且三段均匀 | 机构趋势，健康持续 | **安心持有**：底仓不动，等均线回调加仓 |
| 🔥 翻倍股+前重后轻 | 20d >100% 但 5d <10% | 涨幅已在 10 天前完成，现在停滞 | **不追**：已经走完了。等充分回调或新催化剂 |
| ⚡ 近期爆发 | 5d 占 20d 的 55%+ | 加速赶顶，潜在买盘快速消耗 | **持仓设止盈，不新开**：放量滞涨立刻走 |
| 🟢 V 型反转 | 5d >0 但 d1-10 <0 | 弱转强，从跌转涨 | **回调确认后试仓**：最有价值的路径类型 |
| 🔴 高位逆转 | 20d 正但 5d < -3% | 正在派发，强转弱 | **清仓**：不犹豫，不等反弹 |
| 🟡 减速 | decel_count > accel_count 且 decel_count > 5 | 行情减速 | **减机动仓，留底仓** |
| 🆕 新爆 | only_5d 标的今日涨停多 | 新方向萌芽 | **轻仓试首板**，确认持续再加 |
| 🐌 🔥 **仅40日慢牛** | 40d Top30 但不在 20d Top30 | 长期累积涨幅，但短期不突出——"利通电子型" | **逐个看40日路径**：匀速推进→低吸；前重后轻→等调整 |
| 🟢 **匀速(40d)** | 40d 前后两段均匀，d1-20 ≈ d21-40 | 慢牛趋势健康，持续性强 | **波段持有**：20日窗口看不到的优质标的 |
| 🟡 **后重前轻(40d)** | d21-40 >> d1-20 | 近期加速，慢牛转加速 | **关注买点**：可能正在进入加速段 |
| 📉 **前重后轻(40d)** | d1-20 >> d21-40，近20日动能衰 | 涨幅已兑现，接近停滞 | **不追**：等充分回调后再看 |

> 判断原则：「翻倍」不需要精确 100%，单月 50%+ 即属于"强赚钱效应"，需要在日报中明确标注并分析其驱动因素（资金类型 × 题材阶段）。

### 🆕 生态博弈全景（游资棋盘）

> ⚠️ **这不是"附录"——这是日报的核心分析框架。** 游资的每一次买卖决策，都是在和这个棋盘上的其他人博弈。

日报必须包含「生态博弈全景」节。这不是平铺地"分别看看机构、游资、散户"——而是**站在游资的位置，理解棋盘上每一个棋手的下一步，然后找到自己的最优落子**。

**数据来源**：`period_returns.json` 中的诊断数据 + [[2-wiki/ecology/]] 全部 7 个页面。

#### 生态博弈分析模板

```
## X、生态博弈全景

> 参考 [[2-wiki/ecology/总纲-A股交易者生态]] · period_returns.json 诊断数据

### 🗺 棋盘总览：当前市场谁在吃肉、谁在喝汤、谁在割肉

[一段话概述：当下的赚钱效应流向 + 各类型交易者的处境]

### 🏛 机构：他们在想什么 → 对我意味着什么

| 博弈维度 | 判断 |
|------|------|
| **他们在做什么** | [从大票占比、行业集中度、趋势持续性推断机构行为] |
| **他们接下来会做什么** | [季报窗口、估值位置、排名压力 → 预判加减仓方向] |
| **对我（游资）的信号** | [机构持续买=趋势可靠可以跟 · 机构开始兑现=警惕高位 · 机构在底部建仓=关注回调机会] |
| **我该怎么做** | [具体的：跟趋势/等回调/不碰] |

### 🎯 其他游资：他们在做什么 → 我要不要跟/抢/撤

| 博弈维度 | 判断 |
|------|------|
| **他们在做什么** | [从涨停家数、加速/减速信号、题材轮动速度推断] |
| **赚钱效应在哪** | [哪个方向在加速？哪个方向在减速？有没有新的点火信号？] |
| **情绪到了什么位置** | [高潮（警惕衰竭板）？分歧（等转一致）？冰点（等首板点火）？] |
| **我该怎么做** | [积极参与/选择性参与/收缩休息/轻仓套利——具体到手法] |

### 🤖 量化：他们在搞什么 → 我怎么不被收割反而利用

| 博弈维度 | 判断 |
|------|------|
| **量化特征出现了吗** | [织布机分时？直线拉升后秒砸？高波动但无趋势？] |
| **如果量化主导** | [不追直线、逆向量化做T、只在放量突破时跟] |
| **我该怎么做** | [具体应对策略] |

### 🛡 某队：会不会出手 → 对我仓位的影响

| 博弈维度 | 判断 |
|------|------|
| **某队会动吗** | [什么情况下某队会出手？现在离那个条件有多远？] |
| **如果某队出手** | [托底=加仓机会 · 打压=减仓避险] |
| **我该怎么做** | [提前布局还是等信号确认？] |

### 💰 散户：他们在想什么 → 我的对手盘是谁

| 博弈维度 | 判断 |
|------|------|
| **散户现在是什么状态** | [FOMO追涨？恐慌割肉？还是麻木不动？] |
| **他们的行为可预测吗** | [如果已有很多翻倍股→散户大概率在追→我的卖点就是他们的买点] |
| **我做对手盘还是做队友** | [当前阶段，散户是我的对手（卖给他们）还是队友（一起推）？] |

### ⚠️ 庄家/主控：有没有坑在等着我

| 博弈维度 | 判断 |
|------|------|
| **有没有庄股特征** | [长期横盘小票突然异动？龙虎榜一家独大？] |
| **如果碰到老庄** | [风没来不碰·一进二被爆头不找原因直接走] |

### 🎯 游资作战计划（综合以上所有判断）

| 维度 | 决策 |
|------|------|
| **总仓位** | 0% / 30% / 50% / 70% / 100% |
| **主攻方向** | [哪个方向赚钱效应最强且可持续？] |
| **核心手法** | [趋势持有 / 分歧低吸 / 打板接力 / 波动做T / 休息] |
| **底仓（锁仓不动）** | [什么票？什么条件才卖？] |
| **活仓（做波动）** | [什么票？什么位置吸？什么位置抛？] |
| **机动（快进快出）** | [什么票？什么信号进？什么信号走？] |
| **一键清仓条件** | [什么情况出现，不计成本全部卖？] |
```

#### 生态博弈撰写的铁律

| 原则 | 说明 |
|------|------|
| **游资视角贯穿始终** | 每一个判断都以"对我意味着什么"收尾。不是"机构在加仓"——而是"机构在加仓，所以趋势可靠，我可以跟"。 |
| **二阶思维** | 不是"这个方向好不好"——是"其他游资会不会来做这个方向？散户会不会追？" |
| **数据驱动** | 每个判断都要有 period_returns.json 中的具体数字或盘面信号支撑 |
| **可执行** | 最终输出必须是仓位、方向、手法、具体触发条件——不是泛泛的"注意风险" |
| **认错机制** | 每个操作计划都要写清楚：什么条件出现→我的判断就错了→必须撤退 |

## 零-B、自动化日报（定时任务）

> Claude Code 运行时，系统会在指定时间自动执行 A 股数据刷新 → 分析 → 邮件发送。

### 定时任务一览

| 时间 | 频率 | 任务 | 分析深度 | 邮件 | 归档 |
|------|------|------|----------|:---:|------|
| **9:17** | 周一至周五 | 盘前简报 | 轻量（300-500 字）| 仅重大事件 | — |
| **10:23** | 周一至周五 | 盘中速览 | 轻量（200-400 字）| 仅极端行情 | — |
| **12:27** | 周一至周五 | 午间分析 | 中等（400-600 字）| 仅重要信号 | — |
| **15:07** | **每天** | ★ **完整收盘日报** | **完整 8 维模板** | **每次发送** | → `2-wiki/data/reports/daily/` |
| **20:57** | 周日 | ★ **周度展望 + 月报** | 综合（600-1000 字）| 每次发送 | → `2-wiki/data/reports/weekly/` + `monthly/` |

### 每轮自动化执行流程

**日报（15:07 触发）**：
```
触发时间到 → Claude 自动执行：
  1. python scripts/run_pipeline.py    ← 数据刷新（行情+新闻）+ 选股池
  2. 读取数据源，按 8 维模板写分析      ← 写入 daily_analysis.md（最新版）
  3. 同时保存 dated 副本               ← 写入 2-wiki/data/reports/daily/YYYY-MM-DD.md
  4. python scripts/send_report.py     ← 发送邮件报告
  5. 更新 meta.json                    ← 记录 last_analysis
```

**周报（周日 20:57 触发）**：
```
触发时间到 → Claude 自动执行：
  1. 读取 2-wiki/data/reports/daily/ 中本周全部日报
  2. 综合归纳：趋势演变、情绪周期变化、核心矛盾转移、赚钱效应强弱变化
  3. 对照 Wiki 框架提炼规律性发现
  4. 写入 2-wiki/data/reports/weekly/YYYY-WXX.md
  5. python scripts/send_report.py 2-wiki/data/reports/weekly/YYYY-WXX.md  ← 发邮件（自动识别为「A股周报」标题）
  6. 更新 2-wiki/data/reports/index.md
  7. 如有新规律 → 反哺更新对应概念页
```

**月报（与周报同触发，仅本月最后一个周日执行）**：
```
  判断：今天是否为本月最后一个周日？
    ↓ 是
  1. 读取 2-wiki/data/reports/weekly/ 中本月全部周报（通常 4 篇）
  2. 综合归纳：月度趋势、主线演变、关键节点回顾
  3. 对照 Wiki 框架提炼月度规律
  4. 写入 2-wiki/data/reports/monthly/YYYY-MM.md
  5. python scripts/send_report.py 2-wiki/data/reports/monthly/YYYY-MM.md  ← 发邮件（自动识别为「A股月报」标题）
  6. 更新 2-wiki/data/reports/index.md
  7. 反哺 Wiki：将月度级别的规律发现更新到对应概念页
```

| 修改触发时间 | CronDelete 旧的 + CronCreate 新的 |
| 立即触发一次 | 对 Claude 说「执行一次收盘日报」 |

### 邮件配置（QQ 邮箱）

1. 复制 `scripts/.env.example` → `scripts/.env`
2. 在 QQ 邮箱设置中开启 SMTP，获取授权码
3. 编辑 `.env` 填入：`SMTP_USER`（QQ邮箱）、`SMTP_PASS`（授权码）、`SMTP_TO`（接收邮箱）

### 管线脚本

| 脚本 | 用途 | 用法 |
|------|------|------|
| `scripts/run_pipeline.py` | 串联数据刷新 + 选股池 | `python scripts/run_pipeline.py` |
| `scripts/send_report.py` | 发送报告邮件（日报/周报/月报） | `python scripts/send_report.py [file.md] [--subject 标题] [--test]` |

### 报告归档体系 & Wiki 反哺

```
2-wiki/data/reports/
├── daily/2026-05-27.md    ← 每日收盘日报自动归档
├── weekly/2026-W22.md     ← 每周日从日报综合
└── monthly/2026-05.md     ← 每月初从周报综合
```

**Wiki 反哺机制**：周报和月报在综合时，如果发现持续验证了某个 Wiki 概念的规律，应主动更新对应概念页的「实战证据」章节。例如：

| 发现 | 反哺目标 |
|------|------|
| 连续多日科创50 高位震荡后破位 → [[情绪周期]] 退潮期新增案例 |
| AI 硬件在退潮期仍有局部龙头 → [[龙头认知]] 补充「主线未死」证据 |
| 防御板块连续跷跷板 → [[共振与跷跷板]] 补充退潮期轮动数据 |
| 量能持续萎缩 + 仓位建议降级 → [[仓位管理]] 补充缩量降仓案例 |

反哺时遵循已有规范：原文引用 + 日期背景 + 结果教训，与现有「实战证据」章节格式一致。

### ⚠️ 日报修正同步规则

> 当用户要求修正 `daily_analysis.md`（如对比杰哥复盘后纠错），**必须同步覆盖 `2-wiki/data/reports/daily/` 中对应日期的存档**。两个文件是同一份日报的不同副本（工作副本 + 归档副本），不同步会导致历史回顾时看到旧版本。

```
修正 daily_analysis.md → 改完后立即 → 覆盖写入 reports/daily/YYYY-MM-DD.md
```

### 🔗 概念页更新 → 检查实战手册

> `实战交易手册.md` 引用 10 个概念页，是每日开盘前快速回顾用的。当任何被引用的概念页发生实质内容变更（新章节、新规律、修正框架）时，LLM 必须主动提醒用户：「实战交易手册可能需要同步更新」，并列出变更涉及的概念页。

| 概念页变更触发 | 动作 |
|--------------|------|
| 情绪周期 / 龙头认知 / 仓位管理 / 买卖点 等核心概念更新 | 主动提醒用户检查 [[实战交易手册]] |
| 用户说「检查手册」 | 对照 10 个引用的概念页，逐项核对手册内容是否过时 |

## 二、全部用法一览

### 🟢 Ingest（摄入）— 把新资料消化进 Wiki

| 场景 | 你对我说 | 效果 |
|------|---------|------|
| 处理 inbox 全部 | `处理 inbox` / `ingest` / `消化` | 批量处理 `0-inbox/` 中所有文件 |
| 处理单个文件 | `处理 0-inbox/注意力机制论文.md` | 只处理指定文件 |
| 处理网页 | `把这篇消化了`（浏览器选中内容时） | 处理你正在看的网页 |
| 处理 PDF | `处理这篇 PDF`（拖入 inbox 后） | 读取 PDF 内容并提取要点 |
| 处理视频/播客笔记 | `处理 0-inbox/播客笔记.md` | 处理你手动记的笔记 |
| 跳过某篇 | `处理 inbox，跳过 xxx` | 批量处理时排除指定文件 |

**背后发生的事情**（一次 ingest 可能触及 10-15 个 Wiki 页面）：
```
读取资料 → 和你讨论核心发现
         → 在 summaries/ 创建摘要页面
         → 在 concepts/ 创建/更新相关概念页面
         → 在 syntheses/ 更新或新建跨资料综合页面
         → 更新 index.md 目录
         → 追加 log.md 日志
         → 将原始资料从 inbox/ 移入 sources/
```

### 🔴 复盘对比（Ingest + 日报校准）— 强制联动

> ⚠️ **这是最容易漏的步骤。** 当用户把自己的市场复盘放进 inbox 并要求与 LLM 日报对比时，以下步骤**强制执行**，不可跳过。

| 触发词 | 动作 |
|--------|------|
| **「复盘对比」/「收录并对比」/「处理 inbox 中的复盘并对比日报」** | 完整执行以下三步 |

**强制执行流程**：

```
Step 1: 正常 Ingest
  → 读取 inbox 中的复盘文件
  → 在 summaries/ 创建复盘摘要页
  → 在 concepts/ 更新相关概念页（新知识点）
  → 将原始文件归档到 1-sources/

Step 2: 逐维度对比
  → 对照 daily_analysis.md（LLM 日报）与杰哥复盘
  → 逐维度对比：情绪阶段 / 核心矛盾 / 仓位策略 / 选股池 / 风险判断 / 操作计划
  → 标记差异点（🟢一致 / 🟡部分差异 / 🔴严重偏差）

Step 3: 🔴 更新校准日志（不可跳过）
  → 读取并更新 [[2-wiki/syntheses/日报对比校准日志]]：
     a. 更新「偏差追踪面板」表格（新偏差类型 + 重复次数累加 + 严重度评估）
     b. 追加当日「对比记录」条目（逐维度对比 + 差异分析 + 教训）
     c. 追加「核心偏差总结」表格（问题→校正→是否连续重犯）
     d. 追加「校准后的修正判断」表格
     e. 如有退潮相关误判，跑「退潮确认清单」并记录
  → 追加 log.md 日志
```

> **为什么单独列出**：对比 + 校准日志更新是 ingest 的自然延伸，但因为涉及两个文件（daily_analysis.md + 日报对比校准日志.md），在常规 ingest 流程中容易被遗漏。单独列为强制步骤后，只要用户说「对比」就必须走到 Step 3。

### 🟤 对话存档（Chatlog）

**自动存档**：每次有实质内容的对话结束时，LLM 自动将本次会话摘要追加到 `2-wiki/chatlogs/[日期].md`，无需用户提示。用户也可随时说「保存对话」或「存档」手动触发。

存档内容：
1. 在 `2-wiki/chatlogs/` 创建/更新当日日志文件（如 `2026-05-27.md`）

**按日期归档**：同一天多次对话追加到同一个日文件中，按会话分段。LLM 判断「有实质内容」的标准：执行了文件读写、Wiki 操作、脚本运行、或做出了分析结论。

## 六、常见问题

**Q: inbox 和 sources 的区别？**
> `0-inbox/` 是待处理区，资料放进去等 LLM 消化。消化完毕后，原始资料移入 `1-sources/` 归档。inbox 理论上大部分时间应该是空的。

**Q: 我可以自己写 Wiki 页面吗？**
> 可以，但不推荐。这个系统的核心价值就是 LLM 承担所有维护工作。你直接告诉我要写什么，我来写。你自己在 `2-wiki/` 里写的页面我不会动。

**Q: 怎么避免 Wiki 变得混乱？**
> 定期说「lint」让我做健康检查。我会找出矛盾、孤立页面和过时内容。

**Q: 能同时处理中英文资料吗？**
> 可以。Wiki 页面默认用中文，概念名按你设定的命名规范来。

**Q: 一篇资料被消化后，我还能修改它吗？**
> `1-sources/` 中的原始资料是只读归档。如需修改，在 `0-inbox/` 放入新版本，说「处理 [文件名]，这篇是 [旧文件名] 的更新版」。

## 目录结构

<!-- 下表定义了每个目录的用途和权限边界。关键是：原始资料层只有人类写入，Wiki 层只有 LLM 写入。 -->
<!-- 这种「谁写什么」的清晰边界是整个系统可靠运行的基础。 -->

| 目录 | 用途 | 谁写入 |
|------|------|--------|
| `0-inbox/` | 待处理的原始资料投放区 | 人类 |
| `1-sources/` | 已归档的原始资料（只读） | LLM 处理后移入 |
| `2-wiki/index.md` | Wiki 全页面目录（按分类、带一行摘要和元数据） | LLM 每次 ingest 更新 |
| `2-wiki/log.md` | 按时间顺序的追加日志 | LLM 每次操作追加 |
| `2-wiki/concepts/` | 概念 & 实体页面 | LLM |
| `2-wiki/summaries/` | 单篇资料摘要页面 | LLM |
| `2-wiki/syntheses/` | 跨资料综合、对比、综述 | LLM |
| `2-wiki/topics/` | 主题/领域概览页面 | LLM |
| `2-wiki/chatlogs/` | 对话日志（按日期归档） | LLM（用户说「存档」时） |
| `3-outputs/` | 生成的输出（Marp 幻灯片、图表等） | LLM |
| `attachments/` | 图片 & 附件 | 人类 / LLM |

## 核心操作

<!-- 下面是三个核心工作流。每个都有明确的触发条件和步骤。 -->

### 1. Ingest（摄入）

<!-- Ingest 是把新资料「消化」进 Wiki 的完整流程。 -->
<!-- 触发条件：你把资料放进 0-inbox/ 然后告诉我"处理"或"ingest"。 -->
<!-- 一次 ingest 可能触及 10-15 个 Wiki 页面——这正是 Wiki 复合增长的机制。 -->

当人类将资料放入 `0-inbox/` 并说 "处理" 时：

1. **读取** — 阅读 `0-inbox/` 中的新资料
2. **讨论** — 简要总结关键发现（对话中）
3. **写摘要** — 在 `2-wiki/summaries/` 创建 `[资料名].md`
4. **提取实体/概念** — 在 `2-wiki/concepts/` 创建或更新相关页面
5. **更新 index** — 在 `2-wiki/index.md` 添加条目
6. **追加 log** — 在 `2-wiki/log.md` 追加记录
7. **归档** — 将原始资料从 `0-inbox/` 移入 `1-sources/`

### 2. Query（查询）

<!-- Query 是最常见的操作——你提问，我搜索 Wiki 并给出带引用的综合答案。 -->
<!-- 好的答案不应该只是"说过就忘"，它们应该沉淀回 Wiki 成为新页面。 -->

用户提问时：
1. 搜索 `2-wiki/` 中相关页面
2. 综合答案，附带引用（链接到具体 Wiki 页面）
3. 好的答案可以归档为新页面或更新现有页面

### 3. Lint（健康检查）

<!-- 随着 Wiki 增长，矛盾、孤立页面、过时信息不可避免。 -->
<!-- 定期 Lint 就像代码库的 CI 检查，保持 Wiki 健康。 -->

定期检查：
- 矛盾声明
- 过时信息
- 孤立页面（无入链）
- 缺失概念
- 交叉引用缺口

## 页面约定

<!-- 统一的格式约定让 Wiki 可被程序化解析（Dataview 插件、grep 等）。 -->

### 所有 Wiki 页面必须包含 YAML frontmatter：

<!-- frontmatter 是 Obsidian 的元数据区，Dataview 可以基于这些字段生成动态表格。 -->

```yaml
---
tags: [概念分类]          # 用于 Obsidian 标签系统和图视图
created: 2026-05-27       # 页面创建日期
updated: 2026-05-27       # 最后更新日期
sources:                  # 引用到的原始资料（溯源链）
  - "[[1-sources/xxx]]"
---
```

### 命名规范

- 概念页面：`[概念名].md`（如 `注意力机制.md`）
- 摘要页面：`[来源名] - 摘要.md`
- 综合页面：描述性标题（如 `Transformer 架构对比.md`）

### 链接规范

<!-- Wiki 的力量来自链接密度。每条规范都在防止 Wiki 退化成一堆孤立文件。 -->

- 使用 Wiki-link：`[[概念名]]`
- 首次提及概念时建立链接
- 每篇 Wiki 页面至少有 1 个入链（避免孤立）

## index.md 条目格式

<!-- index.md 是 Wiki 的「目录页」，按分类列出所有页面。 -->
<!-- 在中等规模（数百页面、上百来源）下，这个手工维护的索引比向量检索更可靠。 -->

```markdown
| 页面 | 摘要 | 日期 | 来源数 |
|------|------|------|--------|
| [[2-wiki/concepts/xxx]] | 一句话描述 | 2026-05-27 | 3 |
```

按分类分组（概念 / 摘要 / 综合 / 主题）。

### ⚡ index.md 更新触发清单

> index.md 不仅是 ingest 后更新，以下操作也需要检查并更新 index.md：

| 操作 | 更新什么 |
|------|---------|
| 修改概念页字数/来源数 | 更新 index.md 概念表中该行的字数、来源数 |
| 生成新日报 | 更新 `reports/index.md` 日表 + 主 index.md 数据区 |
| 追加校准日志条目 | 更新偏差追踪面板摘要（如偏差次数变化） |
| 新增/移动/删除任何 Wiki 页面 | 同步 index.md 对应分类表 |
| 周报/月报生成 | 更新 `reports/index.md` 对应表 |
| 概念页发生框架级修改 | 更新 index.md 中该概念的一行摘要 |

## log.md 条目格式

<!-- log.md 是只追加的时间线，记录每一次 Wiki 变更。 -->
<!-- 统一前缀格式 `## [日期] 操作 | 描述` 让 grep 可以精确解析。 -->

```markdown
## [2026-05-27] ingest | 资料标题
- 创建摘要：[[2-wiki/summaries/xxx]]
- 更新概念：[[2-wiki/concepts/yyy]]（新增对比表）
- 更新概念：[[2-wiki/concepts/zzz]]（补充引用）
```

使用一致的 `## [日期] 操作类型 | 描述` 前缀，方便 grep 解析。

### 📦 log.md 归档规则

> log.md 超过 **1000 行**时，LLM 必须主动将旧日志归档：
> 1. 创建 `2-wiki/log-archive/` 目录（如不存在）
> 2. 按月份拆分：将旧条目移入 `2-wiki/log-archive/YYYY-MM.md`
> 3. log.md 只保留当前月 + 上月的条目（约 200-400 行）
> 4. 在 log.md 顶部添加归档提示：`> 📦 更早日志见 [[2-wiki/log-archive/YYYY-MM]]`
> 5. 更新 index.md 目录结构

## 核心哲学

<!-- 下面四条是整个系统的设计理念，来自 Karpathy 的 gist。 -->
<!-- 理解「为什么这么做」比记住「怎么做」更重要。 -->

- **Wiki 是 LLM 的代码库**：Obsidian 是 IDE，LLM 是程序员，Wiki 是持续复合的产物
- **人类负责策展和提问**：选择好资料，提出好问题——这是人类不可替代的价值
- **LLM 处理一切繁琐事务**：不会无聊，不会忘记更新交叉引用，维护负担为零
- **Wiki 是活的**：每次 ingest 都可能触及 10-15 个页面，知识在链接中涌现
